## 🔍 조사 결과: 한국 행정구역 GeoJSON 데이터 (영수 씨가 서칭 요청한 것)

**결론: 완전 무료로 구할 수 있고, 한 번 다운로드하면 프로젝트에 포함시켜서 끝이에요.**

### 추천 데이터 소스 (내 판단 1순위)

**`github.com/southkorea/southkorea-maps`**

- 통계청(KOSTAT) 공식 데이터 기반
- GeoJSON + TopoJSON 형식 (TopoJSON이 용량 훨씬 작음)
- Simplified 버전 제공 → 정밀도 낮추면 파일 크기 대폭 감소
- 라이선스: KOSTAT 공공데이터 → 무료 사용 OK
- 시/도, 시/군/구, 읍/면/동 레벨 모두 있음

**`github.com/vuski/admdongkor`**

- 행정동 경계 파일 (WGS84 좌표계, GPS와 동일)
- 행정구역 변경 이력까지 반영한 정확한 버전
- GeoJSON 직접 제공

### Mapbox 비용 — 걱정 안 해도 돼요

월 50,000 맵 로드까지 완전 무료.
**"맵 로드"는 지도가 처음 뜰 때 1번 카운트, 이후 줌/패닝/레이어 토글/GeoJSON 올리기 전부 추가 비용 없음.**

- GeoJSON 클라이언트에서 처리하면 Geocoding API 호출 없음 → Mapbox 비용 = 지도 렌더링 비용만
- 사용자 50,000명이 하루에 한 번씩 들어와도 무료
- 50,000명 넘어도 1,000 로드당 $5 (= 약 7,000원) → 100만 명이어도 월 약 700만원 수준
- 현실적으로 초기엔 0원

## 고민 10. GeoJSON 파일 크기와 브라우저 로딩 전략

한국 행정구역 GeoJSON 파일들의 실제 크기 추정:


| 레벨     | 폴리곤 수 | 원본 크기 | Simplified 크기 | TopoJSON |
| ---------- | ----------- | ----------- | ----------------- | ---------- |
| 시/도    | 17개      | ~1MB      | ~100KB          | ~30KB    |
| 시/군/구 | ~250개    | ~10MB     | ~1MB            | ~300KB   |
| 읍/면/동 | ~3,500개  | ~50MB     | ~5MB            | ~1.5MB   |

**읍/면/동까지 한 번에 로드하면 무거워요. 전략이 필요해요.**

### 제안하는 로딩 전략

```
앱 첫 실행:
  → 시/군/구 GeoJSON (~1MB) 만 로드 (Simplified)
  → 지도 표시 시작

사용자가 특정 지역 줌 인 시:
  → 해당 시/군/구 안의 읍/면/동 GeoJSON만 추가 로드
  → 예: 서울시 마포구 줌 인 → 마포구 읍/면/동 데이터만 fetch

이 방식의 장점:
  → 초기 로딩 가벼움 (~1MB)
  → 안 가본 지역 데이터는 아예 안 내려받음
  → 자주 가는 지역은 브라우저 캐시로 빠르게
```

### 실제 구현 방법

파일을 시/군/구 코드별로 분리해서 `/geojson/11110.json` (종로구) 식으로 저장.
줌 인하면 해당 코드 파일만 fetch. Next.js의 `/public` 폴더에 넣으면 CDN 처리 자동.

> 답변: 너무 좋아

## 고민 11. Point-in-Polygon 성능 — GPS → 행정구역 변환 속도

사진 GPS 좌표가 "어느 행정구역 안에 속하는지" 계산하는 것.

```
[단순 방법]  좌표 1개 × 시/군/구 폴리곤 250개 = 250번 계산
             사진 1,000장이면 250,000번 계산 → 느림

[최적화 방법]  1. 먼저 Bounding Box로 후보 필터링 (연산 99% 제거)
              2. 후보 폴리곤에 대해서만 정확한 계산
              결과: 1,000장도 1-2초 이내
```

**라이브러리: `turf.js`**

- 정확한 point-in-polygon 계산
- 번들 크기: 필요한 기능만 골라 import 가능 (~30KB)
- Web Worker 안에서 돌리면 UI 블로킹 없음

이건 기술적으로 잘 알려진 방법이라 구현 어렵지 않아요.

## 고민 12. 안드로이드 웹앱에서 갤러리 접근

영수 씨가 "안드로이드는 무조건 빠르고 편하게"를 원해요.

**Chrome on Android (버전 86+)에서 File System Access API 지원:**

```javascript
// 이 코드로 폴더 선택 다이얼로그가 열림
const dirHandle = await window.showDirectoryPicker();
// 사용자가 DCIM/Camera 폴더 선택 → 모든 사진 파일 읽기 가능
// 서버 전송 없이 로컬에서 직접 처리
```

**실제 UX 흐름 (안드로이드):**

1. SnapRoute 웹앱 열기 (Chrome)
2. "사진 폴더 선택" 버튼 탭
3. 파일 매니저 열리고 DCIM/Camera 선택
4. 권한 허용 한 번만 → 이후는 자동
5. 처리 시작 ("1,247장 분석 중...")

**주의할 점:**

- 삼성 인터넷 브라우저는 File System Access API 지원 불안정 → **Chrome 필수** 안내 필요
- 안드로이드 갤러리 앱에서 공유 버튼 → SnapRoute로 보내기도 가능 (단, 한 장씩)

## 고민 15. 색상 팔레트 제안 (영수 씨가 Claude한테 맡겼으니 내가 정함)

미니멀 스타일 + 방문 일수 표현에 어울리는 팔레트:

```
[배경 지도]     연한 회색 계열 (Mapbox Light 스타일)
[미방문 구역]   #E8E8E8 (아주 연한 회색)
[방문 1-2일]    #C8E6C9 (아주 연한 초록)
[방문 3-7일]    #81C784 (연한 초록)
[방문 8-20일]   #388E3C (중간 초록)
[방문 21일+]    #1B5E20 (진한 초록)
```

초록 계열로 정한 이유:

- 지도 위에 파랑은 물(강, 바다)이랑 혼동될 수 있음
- 빨강/주황은 위험 신호처럼 느껴짐
- 초록 = 자연, 여행, 탐험 이미지와 어울림
- 미니멀 흰 배경 지도 + 초록 조합이 깔끔함

나중에 "테마 컬러 변경" 기능 추가 시 쉽게 확장 가능.

## 고민 16. 타임존 처리 구체화

"KST 기준으로 가정"했는데, 실제 EXIF에는 타임존 정보가 없는 경우가 많아요.

**카메라/폰별 EXIF 시간 저장 방식:**

- 삼성 갤럭시: 로컬 시간 (KST) 저장 → 문제없음
- 아이폰: UTC 저장, Offset 별도 태그에 있음 (iOS 13+부터 개선)
- 구형 DSLR: UTC 저장, Offset 없음

**MVP 처리 방안:**

1. EXIF `DateTimeOriginal` 값을 그대로 KST로 간주
2. 해외 사진은 현지 시간이 찍혀있을 텐데, 어차피 날짜 카운팅에 1일 정도 오차
3. 사용자가 나중에 "내 카메라 타임존" 설정 가능하게 하는 기능을 추후에 추가

## 고민 17. 줌 레벨 전환 구체적 설계

줌 레벨에 따라 시/군/구 ↔ 읍/면/동이 전환되는 방식.

**Mapbox 기준 줌 레벨:**

```
전국 뷰:   zoom 5-6  → 시/도 경계만 표시
광역시/도:  zoom 7-8  → 시/군/구 표시, 방문 색칠 + 숫자
시/군/구:   zoom 10+  → 읍/면/동 표시로 전환
```

**전환 시 자연스러운 애니메이션이 중요해요.**

- Mapbox에서 레이어 opacity를 줌 레벨에 따라 자동으로 바꿀 수 있음
- 시/군/구 레이어가 fade out되면서 읍/면/동 레이어가 fade in

## 고민 18. 행정구역 숫자 표시 방식

방문 날짜 수를 행정구역 안에 숫자로 표시할 때:

- **작은 구역**: 숫자가 폴리곤 밖으로 나갈 수 있음 (섬, 작은 동)
- **줌 레벨에 따른 글자 크기**: 줌 아웃 시 글자 다 겹치면 가독성 없음
- **0일 (안 간 곳)**: 숫자를 아예 안 표시? 아니면 회색으로 0?

**내 제안:**

- 방문한 곳만 숫자 표시 (안 간 곳은 숫자 없음)
- 줌 레벨 낮으면 숫자 안 보이다가 줌 인하면 등장 (Mapbox 레이어 minzoom 설정)
- 폴리곤 중심점에 라벨 레이어 별도로 올리는 방식
