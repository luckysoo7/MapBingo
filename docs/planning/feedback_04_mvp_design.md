## ⚠️ 재고 요청: "세션 끊기면 없어져도 됨" — 이거 다시 생각해봐요

feedback_02에서 영수 씨가 "탭 닫으면 없어져도 됨"을 선택했는데,
실제로 어떤 의미인지 체감해봐야 해요.

**현재 결정대로라면:**

```
오늘 사진 1,500장 올려서 지도 완성
→ 탭 닫음
→ 내일 다시 열면 빈 지도
→ 다시 사진 폴더 선택 → 1,500장 또 파싱 (1-2분 대기)
→ 매번 반복
```

이게 괜찮으면 진짜 OK예요. 근데 대부분의 사용자는 이걸 경험하고 "왜 저장이 안 되지?" 생각해요.

**제안하는 중간 옵션:**

사진 원본 저장하는 게 아니에요. **GPS 좌표 + 날짜 데이터만** 브라우저 IndexedDB에 저장.
사진 1만 장이어도 이 데이터는 약 500KB 불과해요.

```
처음 사진 업로드 → 파싱 → 지도 표시 + IndexedDB에 좌표 데이터 저장
다음번 방문 → 즉시 지도 표시 (파싱 불필요)
사진 원본은 여전히 서버에 안 올라가고, 로컬에만 있음
```

장점: 매번 파싱 불필요, 사용성 대폭 향상
단점: 브라우저 IndexedDB에 내 이동 경로가 남음 (보안/프라이버시 의식하는 사람은 불편)
해결: "저장 안 하기" 옵션을 선택지로 주면 됨

>> 답변 (재고할 건지 / 매번 다시 올려도 괜찮은지): 오 너가 말한대로 가자. 저장 기본으로, 안하기는 옵션으로

## 고민 19. MVP 범위 명확하게 확정 ⭐

지금까지 결정된 것들을 바탕으로 MVP에 정확히 뭐가 들어가는지 확정이 필요해요.
아래 내가 제안하는 MVP 범위를 검토해줘요.

### MVP에 반드시 들어가는 것 (없으면 서비스 안 됨)

- [ ] 사진 폴더 선택 (PC 드래그앤드롭 + 폴더 선택, 안드로이드 Chrome 폴더 선택)
- [ ] EXIF GPS + 날짜 추출 (exifr, Web Worker)
- [ ] GPS → 시/군/구 매핑 (GeoJSON + turf.js)
- [ ] 한국 시/군/구 지도 표시 (Mapbox + 행정구역 GeoJSON)
- [ ] 방문 지역 색칠 (로그 스케일, 초록 팔레트)
- [ ] 방문 일수 숫자 표시 (줌 레벨 따라 on/off)
- [ ] 행정구역 클릭 → 통계 카드 팝업 (방문 일수, 사진 장수, 첫 방문일)
- [ ] HEIC 파일 스킵 + 안내 ("3장은 미지원 형식으로 건너뜀")
- [ ] GPS 없는 사진 스킵 + 안내
- [ ] 처리 진행률 표시 ("1,247장 중 832장 분석 중...")

### MVP에 안 들어가는 것 (백로그)

- 읍/면/동 단계 (backlog)
- 스냅샷 공유 (backlog)
- 사진 썸네일 핀 (backlog)
- 위성 지도 (backlog)
- 기간 필터 (backlog)

>> 답변 (빠진 게 있거나 빼고 싶은 게 있으면):좋아. 그런데 mvp를 나중에 클로드 코드가 코드로 구현할때 이거 한번에 하라고 하면 완전 망할 것 같은데, 어떻게 이걸 나눠어서 하나하나 구현할 수 있게 '코드 구현하는 클로드 코드' 에게 일을 시킬까~

## 고민 20. 데이터 모델 (코드 짜기 전에 이게 확정돼야 해요)

앱 안에서 데이터가 어떻게 생겼는지 설계예요.
사진 파싱 후 메모리에 들고 있을 데이터 구조:

```typescript
// 사진 1장당
type PhotoData = {
  id: string          // 파일명 기반 고유 ID
  lat: number         // GPS 위도
  lng: number         // GPS 경도
  date: string        // "2024-03-15" (KST 기준)
  districtCode: string  // 행정구역 코드 "11140" (마포구)
  districtName: string  // "마포구"
  sidoName: string    // "서울특별시"
}

// 집계 결과 (지도에 표시할 것)
type DistrictStats = {
  code: string        // "11140"
  name: string        // "마포구"
  visitDays: number   // 방문 날짜 수 (중복 제거)
  photoCount: number  // 사진 장수
  firstVisit: string  // "2021-03-15"
  lastVisit: string   // "2024-12-01"
}
```

이 구조면 지도 표시, 통계 카드, 나중에 사진 썸네일까지 다 커버 돼요.
크게 이의 없으면 확정할게요.

## 고민 23. 상태관리 + 기술 스택 최종 확정

기술 선택에서 아직 안 확정된 것들이에요.

### 상태관리

사진 파싱 결과(PhotoData 수만 개)를 앱 전체에서 공유해야 해요.

**옵션 A: Zustand** (내 추천)

- 아주 간단한 API, React 초급도 쉽게 쓸 수 있음
- 보일러플레이트 최소
- 번들 크기 작음 (~3KB)

**옵션 B: React Context + useReducer**

- 외부 라이브러리 없음
- 대량 데이터(수만 개) 업데이트 시 성능 문제 가능

**옵션 C: Jotai / Recoil**

- 원자 단위 상태 관리, 세밀한 리렌더링 제어
- 약간 학습 필요

>> 답변 (잘 모르겠으면 내가 Zustand로 확정할게요): 너 추천대로.
>>

---

### 최종 기술 스택 (내 확정 제안)

```
프레임워크:   Next.js 14 (App Router)
언어:         TypeScript (처음엔 낯설지만 배우면 오히려 쉬워짐, 오류를 코드에서 잡아줌)
지도:         Mapbox GL JS
GeoJSON:      southkorea/southkorea-maps (공공데이터, 무료)
EXIF:         exifr
공간 연산:    @turf/boolean-point-in-polygon + @turf/bbox
성능:         Web Worker (EXIF 파싱 + point-in-polygon 연산)
상태관리:     Zustand
스타일링:     Tailwind CSS
배포:         Vercel (무료 플랜)
```

TypeScript를 쓰자고 하는 이유:

- JS 초급자에게 오히려 TypeScript가 도움이 돼요
- 어디서 뭘 전달해야 하는지 IDE가 알려줌 (자동완성, 오류 표시)
- 대규모 데이터(PhotoData 수만 개)를 다루는 코드에서 타입 없으면 나중에 디버깅 지옥

>> 답변 (TypeScript 거부감 있으면 말해줘요, JS로도 가능해요): 좋아 동의해

## 고민 24. 첫 번째 화면 — "빈 상태"에서 뭘 보여줄 건지

사진을 아직 안 올린 첫 방문자한테 어떤 지도를 보여줄지.

**옵션 A: 빈 드래그앤드롭 화면** (지도 없음, 텍스트만)

- 단순하고 명확
- 서비스 설명 잘 해야 함

**옵션 B: 회색 한국 지도 미리 보여주기** (드래그앤드롭 안내와 함께)

- 업로드 전부터 "이렇게 채워질 거야" 예시를 보여주는 효과
- 시각적으로 더 임팩트 있음

**옵션 C: 데모 데이터로 미리 채워진 지도**

- 예시 데이터로 이미 색칠된 지도를 보여주고 "내 사진으로 바꾸기" 버튼
- 서비스 가치를 즉시 전달할 수 있음
- 구현이 약간 더 복잡
