## 고민 0. 웹앱에서 사진 파일 처리 방식 (인터넷 전송 여부) ← 먼저 읽어요

**결론: 사진 파일은 인터넷으로 전송되지 않아요. 브라우저가 로컬에서 직접 읽어요.**

```
[일반 업로드]   사진 → 인터넷 → 서버 → 처리 (우리 방식 아님)
[우리 방식]     사진 → 브라우저가 로컬에서 바로 읽기 → EXIF 추출 → 지도 표시
                       ↑ 인터넷 이동 없음, 수십 기가 사진도 OK
```

HTML5 File API 덕분에, 사용자가 파일/폴더를 선택하면 브라우저가 네트워크 없이 로컬에서 직접 읽어요.
앱이 "파일 시스템 권한 부여 후 로컬 읽기" 하는 것과 원리가 같아요.
EXIF(GPS 좌표 포함)만 뽑으면 사진 원본 데이터는 아무 데도 안 가요.

**그러나 웹앱 vs 앱 간 실질적인 차이가 하나 있어요:**

| 상황 | 웹앱 | 네이티브 앱 |
|---|---|---|
| PC에서 폴더 선택 | ✅ 폴더 통째로 선택 가능 (File System Access API) | ✅ |
| PC에서 자동 스캔 | ⚠️ 사용자가 매번 폴더 선택해야 함 | ✅ 백그라운드 자동 스캔 가능 |
| 폰(Android) 갤러리 접근 | ⚠️ 일부 가능, 제한적 | ✅ 갤러리 완전 접근 |
| 폰(iOS) 갤러리 접근 | ❌ Safari에서 File System Access API 미지원 | ✅ 완전 접근 |
| 수십 기가 처리 | ✅ 원본 전송 없이 EXIF만 읽으므로 OK | ✅ |

**핵심 문제: iOS 폰에서 웹앱은 갤러리에 자유롭게 접근하기 어려워요.**

영수 씨가 "PC + 폰 둘 다" 쓰고 싶다고 했는데,
PC는 웹앱으로 완벽하게 되지만, **폰(특히 iPhone)에서는 웹앱의 한계가 명확해요.**

### 결정 필요 사항 Q0. 폰 지원 방식

## 고민 1. GPS 좌표 → 행정구역 이름으로 변환하는 문제 ⭐ 가장 중요

사진에서 GPS 좌표(위도/경도)를 뽑는 건 쉬운데,
그게 **"서울특별시 마포구"** 인지 **"Tokyo, Shinjuku"** 인지 알아내려면
좌표를 행정구역 이름으로 바꾸는 **Reverse Geocoding** 작업이 필요해요.

방법이 크게 두 가지예요.

### 방법 A: 외부 API 사용 (Mapbox Geocoding, Google Maps API 등)
- 장점: 간단함, 전세계 지원, 정확함
- 단점: **유료**. Mapbox는 월 10만 건까지 무료인데, 사진 1장당 1건 호출이면
  사진 수천 장 업로드 시 한 유저가 수천 건 소비. 유저 많아지면 비용 폭탄 가능.
  Google Maps Geocoding API는 월 40,000건 무료 이후 1,000건당 $5.
- 클라이언트사이드? 이 API들은 서버 API Key 필요 → 키를 숨기려면 서버 필요.
  완전 클라이언트사이드 구조와 충돌.

### 방법 B: 클라이언트에서 GeoJSON으로 직접 계산 (point-in-polygon)
- 전세계 행정구역 경계선 데이터(GeoJSON)를 미리 번들에 포함해놓고,
  좌표가 어느 폴리곤 안에 속하는지 직접 계산.
- 장점: 완전 무료, 서버 없음, API Key 없음
- 단점: **GeoJSON 파일 용량 문제**
  - 국가 단위 전세계 데이터: ~5MB (괜찮음)
  - 한국 시/군/구 단위: ~30MB (무거움)
  - 전세계 시/군 단위: 수백MB (현실적으로 불가능)
- 그래서 어느 단계까지 지원하느냐가 핵심 결정 사항.

### 방법 C: Nominatim (OpenStreetMap 무료 API)
- 완전 무료, 오픈소스
- 단점: 1초에 1건 요청 제한 → 사진 수백 장이면 수백 초 걸림. 사실상 불가능.
  자체 Nominatim 서버 설치하면 해결되지만 서버가 생김.

**→ 내 현재 생각:** 방법 B를 하되, 줌 레벨에 따라 불러오는 데이터를 다르게 가져가야 할 것 같아요.
"한국 안에 있으면 시/군/구 단위 GeoJSON 로드, 해외면 국가 단위만" 같은 방식.
근데 이것도 영수 씨의 결정이 필요해요 (아래 고민 2번 참고).

>> 일단 한국을 우선으로 하고 B로 하자!! 심플하게 나누면 생각보다 할만할 것 같은데..? 로드가 큰가.

## 고민 2. 행정구역 "단계"를 어떻게 설계할 건지

전세계 행정구역 체계가 나라마다 완전히 달라요.

| 나라 | 1단계 | 2단계 | 3단계 |
|---|---|---|---|
| 한국 | 특별시/광역시/도 | 시/군/구 | 읍/면/동 |
| 일본 | 도도부현 | 시/구/정/촌 | - |
| 미국 | 주(State) | 카운티(County) | 시(City) |
| 프랑스 | 레지옹 | 데파르트망 | 코뮌 |

**결정해야 할 것:**

1. 줌 레벨에 따라 단계가 바뀌는 방식으로 갈 건지?
   - 줌 아웃 → 국가 단위 색칠
   - 줌 인 → 해당 나라의 1단계 행정구역 (한국이면 도/광역시)
   - 더 줌 인 → 2단계 (시/군/구)

2. 아니면 단계를 고정할 건지?
   - 전세계 통일: 항상 "나라의 1단계 행정구역"만 표시
   - 단순하지만, 한국에서 "서울 안에서 마포구 vs 종로구"를 구분 못함

내가 추천하는 방향은 **"줌 레벨별 2단계"**예요.
- 줌 아웃: 국가 단위
- 줌 인: 해당 나라의 시/도 단위
- (동 단위까지는 MVP에서 안 해도 됨)
>> 줌레벨 2단게 좋아. 그리고 일단 한국만 하는거야 시/군/구랑 읍/면/동의 큰 두 섹션이 있겠다. 줌 레벨에 따라서.

## 고민 3. "방문 날짜 수" 계산 로직

이게 생각보다 정의가 필요해요.

예시:
- 2024-03-15에 서울 마포구에서 사진 50장 찍음 → **마포구 방문 1일**
- 2024-03-15에 마포구 + 용산구 둘 다 돌아다님 → 마포구 1일, 용산구 1일 **각각 별도 카운트**
- 2024-03-15, 2024-03-16 이틀 연속 마포구 방문 → **마포구 방문 2일**

이 정도면 자연스럽게 느껴지죠? 그런데 **날짜 기준을 어디로 잡느냐** 문제도 있어요.
- 사진의 EXIF 촬영 시간 기준 (타임존이 현지 시간인지 UTC인지 카메라마다 다를 수 있음)
- 해외여행 시 타임존 차이로 날짜가 하루 틀릴 수 있음

→ MVP에서는 일단 **EXIF 시간 그대로** 쓰고, 타임존 이슈는 추후 처리로 넘기는 게 현실적이에요.
> 응 자연스럽네. 근데 날짜 기준은 일단 한국시로 가정하자.

## 고민 4. 색상 인코딩 방식

방문 일수에 따라 색을 얼마나 진하게 할 건지, 기준이 필요해요.

**옵션 A: 선형 스케일** — 1일=가장 연함, 최대일=가장 진함
- 단순하지만 특정 지역만 압도적으로 많이 가면 나머지가 다 비슷하게 보임

**옵션 B: 로그 스케일** — 1→2일 차이보다 10→20일 차이를 작게 표현
- 데이터 분포가 편향됐을 때 더 보기 좋음 (서울은 200일, 제주는 3일 같은 경우)

**옵션 C: 구간별 고정** — 예를 들어
- 1-2일: 연한 파랑
- 3-7일: 중간 파랑
- 8-20일: 진한 파랑
- 21일+: 가장 진한 파랑

→ 내 추천은 **로그 스케일**이에요. 서울/집 근처는 압도적으로 방문 횟수가 많을 텐데,
그게 지도 전체를 압도하면 볼 맛이 없거든요.

색상 팔레트는 단색 계열(파랑 계열, 초록 계열 등)이 깔끔해 보여요.
추후에 영수 씨가 색상 직접 고를 수 있는 커스텀 기능 추가 가능.
>로그로가자. 팔레트는 너가 우선 정하고.

## 고민 5. 전세계 GeoJSON 데이터 용량 문제

행정구역 경계선 데이터를 파일로 갖고 있어야 하는데, 전세계를 커버하면 너무 무거워요.

**현실적인 해결 방법들:**

1. **온디맨드 로딩**: 해당 나라/지역의 GeoJSON만 필요할 때 다운로드
   - 예: 사진들이 한국/일본/프랑스에 있으면 그 3개국 파일만 로드
   - 초기 로딩 가볍지만, 나라마다 별도 파일 관리 필요

2. **Natural Earth 데이터 활용**: 국가 단위는 ~5MB짜리 무료 공개 데이터 있음
   - 1단계 행정구역(시/도)까지 포함한 것도 있음 (~30MB, 압축 시 ~10MB)

3. **Mapbox 벡터 타일 활용**: Mapbox 자체가 행정구역 경계를 레이어로 제공함
   - 좌표 → 행정구역 매핑을 Mapbox API로 처리
   - 이 경우 Mapbox에 종속되지만 용량 문제 해결
   - API 비용 다시 고려 필요

→ **내 현재 결론**: Mapbox 벡터 타일을 활용하는 게 현실적으로 가장 깔끔해요.
GeoJSON 용량 고민이 사라지고, 시각화도 Mapbox에서 직접 처리해주니까요.
단, Mapbox API 호출 방식 설계를 잘 해야 비용을 통제할 수 있어요.
>일단 한국만 하자. 근데 mapbox로 비용이 만원 이상 들까? 그러면 곤란할 것 같은데. 일단 내 사진 20GB 정도만 한다고 쳤을때 말한거야. 그리고 행정구역 데이터 한번 받으면  끝나는거 아닌가? 조금더 나은 방식 있는지 국내 지리정보 등 서칭해봐~

## 고민 6. HEIC (iPhone 기본 포맷) 처리

결정: MVP에서는 HEIC 완전 지원 안 함. 하지만 어디까지 지원되는지 명확히 해야 해요.

**exifr 라이브러리 기준 지원 현황:**
- JPEG: 완전 지원
- PNG: GPS 데이터 보통 없음 (스크린샷 등)
- HEIC: exifr이 부분 지원하지만 **Safari에서만** 제대로 동작, Chrome에서 불안정
- HEIF: HEIC와 유사한 문제

**현실적 시나리오:**
- iPhone 사용자가 사진 앱에서 JPEG로 내보내기 하면 해결됨
- AirDrop으로 Mac에 옮기면 자동으로 JPEG 변환되는 경우도 있음
- 하지만 사용자한테 "JPEG로 변환해서 올려주세요"라고 시키는 건 UX가 나쁨

**MVP 처리 방안:**
- 업로드 시 파일 확인해서 HEIC면 "현재 미지원, JPEG로 변환 후 업로드 요청" 안내
- exifr로 읽기 시도하고 실패하면 해당 파일 스킵 + 몇 장 스킵됐는지 알려주기
>>응응 mvp대로 가자.

## 고민 7. 수천 장 사진 성능 문제

수천 장 EXIF를 한꺼번에 읽으면 브라우저 메인 스레드가 멈춰요 (화면 얼음 현상).

**해결 방법: Web Worker**
- EXIF 읽기 작업을 백그라운드 스레드로 넘기는 방식
- 사용자는 진행률 바 보면서 기다림 ("237 / 1,423장 처리 중...")
- Next.js에서 Web Worker 쓰는 게 약간 설정이 필요하지만 어렵지 않음

이건 MVP부터 넣어야 해요. 안 하면 수백 장만 올려도 브라우저 먹통 됨.

## 고민 8. 스냅샷(인스타 공유) 기술 방법

지도 화면을 이미지로 저장하는 게 생각보다 까다로워요.

**Mapbox의 경우:**
- 지도 자체가 WebGL Canvas로 렌더링됨
- 일반 html2canvas로는 지도 부분이 빈 검정으로 나옴
- **해결책**: Mapbox의 `map.getCanvas().toDataURL()` 메서드 사용 + DOM 요소(숫자 레이블 등) 합성

**단계:**
1. Mapbox Canvas 추출
2. 그 위에 행정구역 숫자 레이블, UI 요소를 canvas에 합성
3. PNG로 다운로드 또는 공유 API 호출

→ 기술적으로 가능하고 레퍼런스도 많아요. MVP에서 구현 가능.
> 응 그래. 약간 후순위 기능이야 이거는.

## 고민 9. 글로벌 지원의 범위

"해외여행도 된다"고 했는데, 어느 수준까지 지원할지 정해야 해요.

**옵션 A: 국가 단위만** (줌 레벨 무관하게)
- 전세계 국가 경계 GeoJSON ~5MB, 가볍고 단순
- "나는 한국/일본/프랑스/미국을 다녀왔다" 수준 시각화
- 국내에서는 시/도 구분이 안 됨

**옵션 B: 국가 + 1단계 행정구역**
- 전세계 시/도급 데이터, 압축 후 ~10-20MB
- "일본 긴키 지방 3일, 간토 지방 5일" 수준
- 용량은 초기 로딩에 부담되지만 캐싱하면 한 번만 로드

**옵션 C: 나라별로 다른 단계**
- 한국은 시/군/구까지, 해외는 국가 단위만
- 한국 사용자 타겟에 최적화된 방식
- 가장 현실적이고 용량/성능 균형이 좋음

→ **내 추천: 옵션 C** — MVP는 한국 시/군/구 + 해외 국가 단위. 추후 해외도 시/도 단계 추가.
> 그래 일단 한국먼저 하자 위에서 말한 대로! 고민 1,2 참고.
