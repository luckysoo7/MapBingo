# MDplanner — 진행 상황 UX 개선

> 웹앱 사용 중 로딩 지연에 대한 피드백이 없어서 사용자가 앱이 멈췄는지 모름.
> 사전 경고 + 단계별 진행 표시 + 지도 실시간 색칠 유지 구현.

---

## 현재 상태 (코드 파악 결과)

- `ProgressBar.tsx` 존재: 하단 중앙 `w-80` 카드, `parsing` 중 N/N 표시됨
- `UploadOverlay.tsx`: `collecting`/`parsing` 중엔 숨겨짐 → **지도는 이미 이 시점에 보임**
- `collecting` 단계: 파일 개수 실시간 카운트 가능 (for await 루프에서 이미 `fileCount` 추적 중)
- 단, store에 `collectingCount` 상태 없어서 UI에 표시 못 함
- 사전 경고(파일 많으면 느림) 텍스트 없음

## Q1. 사전 경고 — 어디에, 어떻게 표시?

폴더 선택 전에 "파일이 많으면 오래 걸린다"는 안내.

선택지:

- A. **UploadOverlay 카드 하단 힌트 텍스트** (항상 표시, 비침습적)
- B. **폴더 선택 직후 확인 모달** ("1,234개 파일이 감지됐습니다. 계속하시겠어요?")
- C. **파싱 시작 후 처음 N초 동안 배너 표시**

추천: **A** — 모달은 흐름을 끊고 귀찮다. 카드 하단에 작은 힌트 텍스트가 가장 무난함.
예시 텍스트: `"사진이 많으면 수 분 걸릴 수 있어요 · 파일은 기기 밖으로 전송되지 않아요"`

> 답변: 안드로이드 media picker툴 한계로 인해 사진 처리에 최대 수십분 소요 가능하다고 안내해

## Q2. 단계 레이블 — 어떤 단계를 어떻게 표기?

현재 status 흐름: `idle → collecting → parsing → done`

각 단계를 사용자에게 어떻게 보여줄지.

선택지:

- A. **단계 번호 포함**: "1/2 파일 수집 중", "2/2 EXIF 파싱 중"
- B. **레이블만**: "파일 수집 중 · 1,234개 발견", "EXIF 파싱 중 · 1,234 / 5,678"
- C. **최소**: 현재처럼 텍스트만, 대신 파일 수 카운트만 추가

추천: **B** — 단계 번호는 불필요한 복잡도. 레이블 + 숫자만으로 충분히 명확.
`collecting`: "파일 수집 중 · 1,234개"
`parsing`: "EXIF 파싱 중 · 1,234 / 5,678장 (21%)"

> 답변: B로

## Q3. ProgressBar UI 위치/크기 — 지도를 얼마나 가려도 되나?

현재: 하단 중앙 `w-80` (320px) 흰 카드, `shadow-lg`.

선택지:

- A. **현재 유지** — 크기 그대로, 텍스트만 개선
- B. **더 슬림하게** — 높이 줄이고, 반투명 처리 (지도가 투과되어 보임)
- C. **좌하단 작은 칩** — 지도 우측 대부분은 완전히 노출

추천: **B** — 반투명 (`bg-white/88 backdrop-blur-sm`) 처리하면 뒤에 지도가 어렴풋이 보여서
"지도가 색칠되고 있다"는 느낌 유지. `w-72` 정도로 약간 슬림하게.

> 답변: B

## Q4. collecting 단계 파일 카운트 표시

`showDirectoryPicker` 사용 시, `for await` 루프에서 파일을 하나씩 발견하는 방식.
현재 수집 단계에서 파일 수가 얼마나 쌓이는지 화면에 안 보임.

구현 방법: `usePhotoUpload`에서 N개마다 store에 `collectingCount` 업데이트.

추천: **표시함** — "파일 수집 중 · 3,421개..." 처럼 숫자가 올라가는 걸 보여주면
사용자가 "앱이 멈춘 게 아니구나"를 알 수 있음. 100개마다 업데이트(UI 과부하 방지).
